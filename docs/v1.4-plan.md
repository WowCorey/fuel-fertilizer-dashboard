# v1.4 plan — "Verify and fill the gaps" (draft)

Status: draft for review. No code in this PR.

Goal: widen verified data coverage, make hand-keyed entry safe and routine,
and expose per-card freshness honestly in the UI. Three independent tracks,
each delivered as one or more small PRs.

Tracks are strictly sequential: Track 1 must be green in CI before Track 2
starts; Track 2 before Track 3. Opportunistic items ship only after all three
are merged.

---

## Track 1 — New programmatic fetchers

Branch name convention: `live-data-<source-id>` — one PR per source.
Order: (a) cheapest and most transparent first (EIA fuel retail), (b) then
RBA, (c) then the four ABS HS-31 subcategories as a single PR because they
share one fetcher, (d) Pink Sheet last because of the rights question below.

Each entry lists `id`, `publisher`, `fetch_url`, `update_cadence`, `rights`
and any concerns. Full sources.yml fields (canonical_url, rights_url,
citation, reuse_notes, used_by) are filled per the registry contract.

### Per-PR validation checklist

Every Track 1 PR description carries:

- `python3 scripts/validate_data.py` (expect exit 0, no new errors).
- `python3 scripts/fetch_data.py --check` (programmatic URLs only).
- `python3 scripts/fetch_data.py --only <new_id>` (produces envelope,
  `status: ok`, `last_data_point` present).
- Rendered screenshot of the dashboard card for that series, or the DOM
  snippet if screenshots are not practical in the sandbox.

No PR lands red in CI and no PR touches more than one new source (the four
HS-31 subcategories count as one because they share one fetcher).

Branch: `tooling-manual-entry-cli`.

```
  - last_data_point: 2026-04-07
  + last_data_point: 2026-04-21
  + values[+2]
```

On save, the CLI prints the full written path and a minimal diff of what
changed (`last_data_point` delta, count of values added).

### Flags

| Flag | Behaviour |
|---|---|
| `--source <id>` | skip the picker; abort if not `fetch: manual` |
| `--since <YYYY-MM>` | only prompt for months missing from `values[]` |
| `--force` | allow overwriting an existing verified point (default refuses) |
| `--dry-run` | write to stdout instead of `data/manual/<id>.json` |

### Envelope-writing rules

- `status`: `"ok"` after first successful save.
- `manual_entry`: `true` (always).
- `retrieved_at`: `datetime.now(timezone.utc).isoformat(timespec="seconds")`.
- `last_data_point`: max `t` across existing + new values, as YYYY-MM-DD.
- `values`: merged list — existing values preserved, new values appended,
  sorted by `t` ascending, duplicates by `t` refused without `--force`.
- `source_name`, `source_url`, `source_id`, `series_id`: derived from
  `data/sources.yml` entry, not re-asked.
- `notes`: merged — existing notes retained, new per-point notes appended
  with a date prefix if provided.

Before writing, the CLI runs `scripts/validate_data.py` against a
temporary file in a tempdir and refuses to save if validation fails. This
is a hard gate, not a warning.

### Tests

`tests/test_enter_manual.py` covers the envelope-writing logic (not the
prompts). Uses the built-in `unittest` module — no new deps. Coverage:

- new envelope creation (no prior file)
- append to an existing `ok` envelope
- refuses to overwrite an existing verified `t` without `--force`
- refuses to save when the resulting envelope fails the validator
- `--since` filters out already-present months
- `retrieved_at` / `last_data_point` shapes are correct

### README addition

Short subsection under "Data pipeline":

```
### Hand-key a value

Use scripts/enter_manual.py — it picks a manual source, prompts for each
value with date and unit confirmation, appends to the envelope, and only
writes if scripts/validate_data.py passes. Never overwrite a verified
value without --force.
```

## Track 3 — Per-card freshness pill

Branch: `ui-freshness-pill`.

### States and thresholds

The pill is derived from the envelope alone. Thresholds come directly from
`STALE_GRACE_DAYS` in `scripts/validate_data.py` so the UI and validator
agree.

| envelope state | cadence window | pill label | token |
|---|---|---|---|
| `status === 'ok'` and `last_data_point` within cadence grace | within | Verified | `--surplus` (teal #0F766E) on `--surplus-tint` |
| `status === 'ok'` but `last_data_point` older than grace | stale | Verified · N days old | `--caution` (amber #B45309) on `--caution-tint` |
| `status !== 'ok'` | n/a | Awaiting source | `--ink-3` (#6B7280) on `--paper-sunk` |

- All three background/text pairs already meet WCAG AA on paper. No new
  colours.
- `N days old` uses tabular figures (already global).
- No iconography. No emoji. Plain English.

### API

Extend the existing envelope-aware branch in `MetricCard` and `ChartCard`
only — single shared helper, no duplicate logic:

```js
// ui_kits/shared/data-loader.js — new utility
window.FR.freshness = function(env, sourceRegistryCadence) {
  // returns { state: 'verified'|'stale'|'awaiting', label: string,
  //          daysOld: number|null }
};
```

`sourceRegistryCadence` is looked up once at load time in each dashboard
by extending `FR.load()` to return an object of `{ envelope, cadence }`
per id. Dashboards pass `env` as before; no downstream change to callers.

### DOM

```html
<span class="pill pill--verified"
      aria-label="Verified — last data point 12 April 2026">
  Verified
</span>
```

- The pill is static content on a static page. It has no `role="status"`
  because there is nothing to announce: the value does not change after
  page load.
- The pill is non-interactive, has no tab stop, and exposes its full text
  via `aria-label`.
- `prefers-reduced-motion`: no transitions on the pill at all, so nothing
  to suppress.
- CSS lives in `ui_kits/shared/styles.css` next to the existing `.pill`
  rules. Variants use existing tokens.

### Preview

`preview/` currently has reference cards for each token group. Three rows
are added to the existing `cards.html`:

- Verified pill — fresh envelope
- Verified · N days old — stale envelope (synthetic fixture, documented
  as a preview-only stub, not checked into `data/`)
- Awaiting source — unavailable envelope

### Non-goals

- Not changing any card layout.
- Not changing any page copy.
- Not introducing a build step or new runtime dependency.
- No dark mode.

## Opportunistic items (only after tracks 1–3 merge)

### CI auto-open an issue on fetch failure

In `.github/workflows/refresh-data.yml`, on `--check` failure:

```yaml
- name: Open issue on fetch failure
  if: failure()
  uses: actions/github-script@v7
  with:
    script: |
      const title = `Programmatic source check failed: ${process.env.FAILED_ID}`;
      const existing = await github.rest.issues.listForRepo({
        owner: context.repo.owner, repo: context.repo.repo,
        state: 'open', per_page: 100,
      });
      if (existing.data.some(i => i.title === title)) return;
      await github.rest.issues.create({
        owner: context.repo.owner, repo: context.repo.repo,
        title, body: `Run: ${context.serverUrl}/${context.repo.owner}/${context.repo.repo}/actions/runs/${context.runId}`,
      });
```

Dedupe by title. Default `GITHUB_TOKEN` is sufficient.

## Hard rules

Reproduced from the brief to make the contract visible in this plan:

- Never invent, interpolate or estimate a value.
- Never mark a manual envelope `status: "ok"` without a real
  `last_data_point` and `retrieved_at`.
- Never add a source without a clear `rights` line.
- Every PR passes `scripts/validate_data.py` and the existing CI.
- No new runtime deps beyond `pyyaml` and `requests`.
- No dark mode. No emoji. No gradients. No fonts outside
  `colors_and_type.css`. Australian English.
- Plain English on any user-facing string; expand acronyms on first use.
